# 13 вЂ” Parameters, Bounds and Constants (single reconciled table)

**Status: normative component specification. Supersedes the corresponding sections of BACKEND_PLAN.md/FRONTEND_PLAN.md** (BE В§21 in full; every bound, constant or default restated anywhere else in either source).

**Boundary.** This document is the **single source of truth for every tunable parameter, kernel constant, storage bound and frozen protocol constant**. Other component documents reference these values and MUST NOT restate different ones (quoting for readability is permitted when marked *(normative value: В§13)*). Chain-identity constants and the integration surface are jointly owned with [02](02-integration-contract.md) (D-17); where both documents state a value they are byte-identical by construction of the D-2 freeze.

Normative language: RFC 2119. Blocks at 6 s (14,400/day); USDC 6 decimals, VIT 12 decimals; `Fixed` = `FixedU64` (1e9 scale) at API boundaries.

**Reading rules (X-11e/X-11h вЂ” no hardcoding, anywhere):**

1. **K** = kernel constant: compile-time in `futarchy-primitives`, no storage representation, changeable only by a Wasm upgrade that the attestation regime surfaces ([09](09-execution-upgrades-and-rollout.md)). **Boundary of the primitives home:** every K-marked numeric value in this document вЂ” the LMSR domain bound and error-bound constants included, which `futarchy-fixed` imports and asserts rather than re-declares вЂ” lives in `futarchy-primitives::kernel`; sibling crates MUST import it, never re-declare it. Two carve-outs: per-pallet **storage-bound arguments** (В§4) live with their owning pallets ([02 В§7](02-integration-contract.md)), and pallet-internal implementation constants with no row in this document need no primitives home. Contract-surface **types** вЂ” every type named in 02 В§В§2вЂ“7, event-field enums such as `MarketKind` included вЂ” likewise live in `futarchy-primitives` (02 В§2).
2. Everything not marked K lives in `pallet-constitution::Params` as a typed record `{ value, min, max, max_delta_per_decision, cooldown, last_change, class, kernel_bounded }`; min/max/max-delta/cooldown/class are genesis-fixed for kernel-bounded keys and META-amendable within compile-time meta-bounds otherwise.
3. **Every value in this document вЂ” K constants included вЂ” is machine-readable by clients** via the runtime **constants API** (metadata) or the named storage item, per the [02](02-integration-contract.md) contract. The frontend MUST read, never hardcode (this closes X-11h's backend side: there is no value the FE re-checks that the chain does not expose).
4. Every default is a **simulation hypothesis** unless marked frozen; Phase 0вЂ“3 calibration obligations are tagged *sim-gated*.
5. `ProposalClass::Emergency` is deleted (D-7): no row in this document carries an EMERGENCY value, and none may be added without reopening D-7.
6. **`ParamKey` encoding (canonical).** The on-chain `ParamKey` ([02](02-integration-contract.md) В§2, `[u8; 16]`) is the row's dotted name, UTF-8-encoded and zero-padded to 16 bytes. Names longer than 16 bytes carry an explicit canonical key, written `key:` in their row вЂ” truncation is forbidden (the encoder rejects over-long names). **Per-class rows** (`dec.delta`, `dec.sigma`, `dec.v_min`, `gate.v_min`, `prop.bond`, `pol.b`, `exec.timelock` в†’ base `exec.lock`, `trs.proposer_reward` в†’ base `trs.reward`) materialize as four keys with the class suffixes `.param` / `.trs` / `.code` / `.meta`. `gate.v_min` is per-class because its default and both its bounds are expressed as multiples of `dec.v_min`(class) вЂ” a single key could not carry a class-derived value (added 2026-07-20, SQ-194).
7. **Record classes and the view projection.** `ParamRecord.class` uses the six-class set {PARAM, TREASURY, META, CONST, entrenched, META+values}; `ParamView.class: ProposalClass` ([02](02-integration-contract.md) В§4) projects CONST and entrenched onto `Constitutional` and META+values onto `Meta`. Rows whose min or max is a kernel floor/ceiling are marked `kernel-bounded`: per rule 2 their **entire** governance-metadata tuple (min/max/max-О”/cooldown/class) is genesis-fixed and `amend_registry` refuses them outright; all other rows are META-amendable within the compile-time meta-bounds (the amendment must keep `min в‰¤ value в‰¤ max`, preserve the value kind, and keep `cooldown в‰¤ 8` epochs). The kernel-bounded set is, normatively and exhaustively (per-row K/floor/ceiling markers plus the В§1 safety rationale; conservative where a bound is cross-key-derived or marked "never lowered"/"down only"): `att.window`, `code.spacing`, `dec.delta.code`, `dec.delta.meta`, `dec.delta.param`, `dec.delta.trs`, `dec.sigma.code`, `dec.sigma.meta`, `dec.sigma.param`, `dec.sigma.trs`, `dec.window`, `epoch.length`, `exec.grace`, `exec.lock.code`, `exec.lock.meta`, `exec.lock.param`, `exec.lock.trs`, `gate.eps`, `gate.p_max`, `intake.slash_pct`, `iss.inflation`, `keeper.budget`, `ledger.archive`, `ledger.min_split`, `ledger.pos_dep`, `mkt.kappa`, `orc.bond_bps`, `orc.n_min`, `orc.window`, `pol.budget_epoch`, `res.fail_thr`, `res.probe_amount`, `res.probe_int`, `res.probe_to`, `res.recover_thr`, `trs.cap_180d`, `trs.cap_30d`, `trs.cap_proposal`, `welfare.thC_lo`, `welfare.thS_lo`, `wt.quorum`. Cross-key couplings that a static record cannot express (`dec.sigma в‰¤ Оґ/2`, `gate.eps в‰¤ p_max/2`, `welfare.wP + wA = 1`, the classless Baseline book grading at the `dec.v_min.trs` floor вЂ” [05](05-welfare-and-decision-engine.md) В§5.2, SQ-232) bind at the consuming engine; the seeded static bounds are conservative over-approximations. **The `gate.v_min` coupling is the exception and binds earlier** (corrected 2026-07-20, SQ-194): `0.1 Г— dec.v_min(class)` is its *default*, not an invariant вЂ” the row admits the band `[Г—0.05, Г—0.5]` вЂ” and that band, evaluated against the **live** `dec.v_min(class)`, is enforced **at the amendment boundary** jointly over the pair, so an amendment of either `dec.v_min.<class>` or `gate.v_min.<class>` that would carry the pair outside it is refused rather than left for a consumer to reconcile. The same relation is asserted in `try-state`. Two constraints therefore bind a `gate.v_min` amendment and the admissible set is their **intersection**: this live coupling band, which is the standing invariant, and the record's own `min`/`max`, which are ordinary META-amendable metadata seeded at genesis from the same multiples of the *then-current* `dec.v_min`. The seeded bounds coincide with the band at genesis but do not track later `dec.v_min` amendments, so the `Г—0.05` / `Г—0.5` columns of the В§1 row state the coupling invariant, not a promise about the stored metadata.
8. **The `Unit` column is human/display notation, not `ParamView` encoding.** The В§1 table's Default/Hard min/Hard max/Max О” figures are written in the human unit printed in that column. By contrast, [02 В§4](02-integration-contract.md) `ParamView.value/min/max` carry the raw inner scalar of the stored `ParamValue`: `Fixed` on the 1e9 grid, `Perbill` in parts per billion, `Percent` as integer percent, and integer/balance variants in their native scalar. In particular, converting raw `Perbill` to displayed basis points divides by **100,000**, floored; market-fee recomputation uses the frozen `Market::Fee` bps metadata constant and cross-checks it against that raw projection.

**Existing-chain introduction of new rows (normative).** Adding a row to the genesis `Params` template is not an upgrade path: any runtime that first consumes a newly introduced key MUST carry a versioned constitution-state migration that inserts the complete bounded record on existing chains without overwriting a lawful value already present. This requirement applies to `ops.probe_rate` and `ops.probe_fee`; their first production runtime seeds both records by migration as well as by genesis, and its try-runtime checks prove the records are present and valid after upgrade.

Safety rationale (row-wise, carried forward): kernel floors/ceilings exist so no captured decision sequence can walk a defense to zero вЂ” Оґ cannot reach 0, windows cannot reach one block, Оє cannot open flash manipulation, timelocks/grace cannot vanish, p_max cannot exceed 0.10, guardian scope cannot grow, the intake slash fraction cannot reach 0, and the Ask-scaling `2В·InCapPrize` term of `dec.v_min` is kernel, not a key.

**Rows without a seeded `Params` record (normative; amended 2026-07-20, SQ-158).** Not every В§1 row materializes as a genesis-seeded record; two shapes are exempt, and reading rule 2 binds every other row.

1. **Entrenched, kernel-scoped rows with no amendment path.** `grd.bond` вЂ” an `entrenched` row whose bound is scope-K and which carries neither a max-О” nor a cooldown вЂ” is carried as a **kernel constant exposed through pallet metadata** rather than as a `Params` record: a record bearing governance metadata would misdescribe a value that cannot be governed. Reading rule 3 is satisfied for it by the constants API rather than by a storage item. This exemption covers the **bond only**. The guardian *allowances* sharing its row are kernel constants that are **not** individually metadata-exposed, so rule 3 is not currently satisfied for them; their exposure is unresolved and is not ratified here.
2. **Rows that gate nothing and have no number yet.** A row carrying **no dispatch-path limit of its own** вЂ” a pure economic, operating-budget or diagnostic value that no extrinsic checks against вЂ” may remain unseeded while its default is still `[VERIFY]`- or *sim-gated* under rule 4; reading rule 2 binds it **from adoption onward**, at which point it is seeded like any other. The tag is **not** itself the exemption, and this is the load-bearing half: a tagged row whose value *does* gate a dispatch or bound is seeded regardless вЂ” which is why the phase-3 exposure caps, the four XCM trader rates and the coretime-quote keys all carry live records despite carrying tags, and why the sentinel note below can rely on the phase-3 caps having seeded hard maxes.

   **Derivation is not an exemption.** Where a value is computable today and the implementation nonetheless derives it at its consuming engine while the В§1 row still advertises an amendment path вЂ” a max-О”, a cooldown, an amendable class вЂ” the missing record is a **defect**, not an exempt shape: the row promises a governance lever that derivation silently removes. (A row still awaiting its first number is the case above, not this one.)

**Unbounded sentinel (normative; amended 2026-07-20, SQ-196).** The "unbounded sentinels" of [09 В§5.2](09-execution-upgrades-and-rollout.md) are the **maximum representable value of the key's `Balance` type**, which is also the seeded hard max of `phase3.tvl_cap` and `phase3.dep_cap` вЂ” so arming one is an ordinary amendment of the row to its own ceiling, requiring no distinguished value and no separate mechanism. (The phase at which each is installed is owned by [09](09-execution-upgrades-and-rollout.md), not by this row.) The two sentinels are **independent, and each disables only its own check**: `phase3.dep_cap` at the sentinel retires the per-account cumulative meter вЂ” which then neither reads nor writes that cumulative вЂ” while `phase3.tvl_cap` at the sentinel lifts only the global-issuance admission test. A bounded cap on either axis continues to reject inflows that breach it, so containment disappears entirely only when **both** sentinels are installed. Entries already recorded are **retained, never reaped**; they stay visible to `try-state` and bind again unchanged if a bounded cap is later restored.
